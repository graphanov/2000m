# 2000m Driver Protocol v0

Protocol id: `2000m.driver.v0`

2000m scores produced Rust SkiFree-homage games through a deterministic, headless process
boundary. A produced game ships a driver executable that reads newline-delimited JSON commands
from `stdin` and writes exactly one newline-delimited JSON response to `stdout` for each command.
The conformance suite never imports game code in-process.

## Transport rules

- Encoding: UTF-8 JSON Lines (`\n` terminated).
- One command in, one response out. Drivers must not write non-JSON text to `stdout`.
- Diagnostic text may be written to `stderr`; it is ignored by scoring unless the process fails.
- One `step` command is exactly one deterministic logic tick. Wall-clock time, frame rate, timers,
  random OS state, networking, and user input outside the command stream must not affect logic.
- All numeric values are finite JSON numbers. Implementations should use deterministic integer,
  fixed-point, or otherwise reproducible arithmetic.
- Unknown command or malformed input responses should be `{"ok":false,"error":"..."}` and should
  not crash the driver.

## Manifest

A produced game root must contain `2000m.json`:

```json
{
  "protocolVersion": "2000m.driver.v0",
  "driver": {
    "command": "cargo",
    "args": ["run", "--quiet", "--bin", "driver"]
  },
  "language": "rust"
}
```

The conformance runner resolves the manifest path from the produced-game directory, starts the
command with that directory as the working directory, sends the protocol commands below, and kills
the process at the end of each check.

## Commands

### `init`

Starts a new deterministic game from a seed.

```json
{"cmd":"init","seed":42,"config":{}}
```

Fields:

- `cmd`: literal `"init"`.
- `seed`: signed integer. Same seed plus same command sequence must produce the same GameState
  stream across separate driver processes.
- `config`: optional object. Drivers should ignore unknown keys. The conformance suite may use
  the following v0 keys to make isolated mechanics testable without an in-process API:
  - `slopeWidthM`: positive number; requested horizontal wrap width, centered around `x=0` when
    the game has no stronger internal convention.
  - `scenario`: optional string, one of:
    - `"collision-tree"`: make a tree/stump collision reachable quickly by normal protocol input.
    - `"ramp"`: make a ramp jump and landing reachable quickly by normal protocol input.
    - `"monster-contact"`: make monster contact reachable quickly after init.

Scenario support is not a separate score track; it is a protocol-level way to expose mechanics to
fixed-tick black-box tests. A driver that implements the mechanics naturally may also pass without
special-casing these hints if the checks can observe the behavior through GameState.

Response:

```json
{"ok":true,"state": { /* GameState */ }}
```

### `step`

Advances the deterministic logic by one tick using an input sample.

```json
{"cmd":"step","input":{"steer":1,"boost":false,"jump":false}}
```

Input fields:

- `steer`: integer `-1`, `0`, or `1` for left, neutral, right.
- `boost`: boolean. Corresponds to the SkiFree F-key accelerate mechanic.
- `jump`: boolean. Requests jump/trick input where implemented.

Response: `{"ok":true,"state":<GameState>}`.

### `state`

Returns the current state without advancing the tick.

```json
{"cmd":"state"}
```

Response: `{"ok":true,"state":<GameState>}`. The returned `tick` must equal the previous state.

### `reset`

Starts a new deterministic game in the same process.

```json
{"cmd":"reset","seed":42}
```

Response: `{"ok":true,"state":<GameState>}`. `distanceM`, `style`, `monster`, `gameOver`, and
`tick` must return to their initial values. The post-reset stream for a seed must match a fresh
`init` stream for the same seed and command sequence.

## GameState

Every successful command returns a complete state object:

```json
{
  "skier": { "x": 0, "y": 0, "speed": 0, "mode": "skiing" },
  "distanceM": 0,
  "style": 0,
  "obstacles": [
    { "type": "tree", "x": 12, "y": 96 }
  ],
  "monster": null,
  "gameOver": false,
  "tick": 0
}
```

Required fields:

- `skier.x`: horizontal skier coordinate in deterministic logic units/meters.
- `skier.y`: downhill/world coordinate in deterministic logic units/meters.
- `skier.speed`: current downhill speed in meters per tick or deterministic equivalent.
- `skier.mode`: one of `"skiing"`, `"crashed"`, `"airborne"`, or `"eaten"`.
- `distanceM`: cumulative downhill distance in meters. It must strictly increase while skiing and
  not increase while a crash halts forward motion.
- `style`: numeric style score.
- `obstacles`: deterministic visible/relevant obstacle stream. Each obstacle has:
  - `type`: one of `"tree"`, `"bigtree"`, `"stump"`, `"mogul"`, `"rock"`, or `"ramp"`.
  - `x`, `y`: deterministic coordinates in the same logic coordinate space as the skier.
- `monster`: either `null` or an object with:
  - `x`, `y`: deterministic coordinates in the same coordinate space as the skier.
  - `mode`: one of `"chasing"`, `"eating"`, or `"fleeing"`.
- `gameOver`: boolean terminal flag.
- `tick`: non-negative integer logic tick. `init` and `reset` return `0`; each `step` increments
  by exactly `1`; `state` does not increment it.

## Determinism requirement

For any seed and any command/input sequence, two independent driver processes must emit identical
GameState streams after canonical JSON serialization. Seeded random generation must be derived
only from the protocol seed and deterministic state. The conformance suite checks this before the
16 acceptance criteria.

## Error response

Drivers should return errors as JSON rather than panic:

```json
{"ok":false,"error":"unsupported command"}
```

An error response, invalid JSON, EOF, timeout, or process crash causes the active conformance
check to fail.

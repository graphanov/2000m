# 2000m Driver Protocol v1

Protocol id: `2000m.driver.v1`

## Changes from v0

- **Quality metrics** in GameState (optional, for Tier 2-3 scoring)
- **New commands**: `profile`, `replay`, `challenge`
- **New GameState fields**: `quality`, `events`, `replayChecksum`
- **28 acceptance criteria** (up from 16) across 3 tiers
- **Composite scoring** replaces binary passCount

## Transport Rules

Unchanged from v0:
- UTF-8 JSON Lines (`\n` terminated)
- One command in, one response out
- Diagnostic text to stderr only
- One `step` = one deterministic logic tick
- All numeric values are finite JSON numbers
- Unknown commands â†’ `{"ok":false,"error":"..."}`

## Commands

### `init` (unchanged from v0)

```json
{"cmd":"init","seed":42,"config":{}}
```

Response: `{"ok":true,"state": <GameState>}`

### `step` (unchanged from v0)

```json
{"cmd":"step","input":{"steer":1,"boost":false,"jump":false}}
```

Response: `{"ok":true,"state":<GameState>}`

### `state` (unchanged from v0)

```json
{"cmd":"state"}
```

Response: `{"ok":true,"state":<GameState>}`

### `reset` (unchanged from v0)

```json
{"cmd":"reset","seed":42}
```

Response: `{"ok":true,"state":<GameState>}`

### `profile` (NEW in v1)

Returns performance metrics collected over the last N ticks.

```json
{"cmd":"profile","window":100}
```

Fields:
- `window`: number of ticks to aggregate (default: all ticks since init/reset)

Response:

```json
{
  "ok": true,
  "metrics": {
    "windowTicks": 100,
    "avgTickNanos": 2300000,
    "maxTickNanos": 8100000,
    "p95TickNanos": 4200000,
    "p99TickNanos": 6800000,
    "totalAllocations": 0,
    "peakMemoryBytes": 45000000,
    "collisionChecksTotal": 1523,
    "collisionHitsTotal": 42
  }
}
```

Required fields:
- `windowTicks`: number of ticks in the measurement window
- `avgTickNanos`: mean wall-clock time per tick in nanoseconds (measured by the driver internally)
- `maxTickNanos`: worst-case tick time
- `p95TickNanos`: 95th percentile tick time
- `p99TickNanos`: 99th percentile tick time
- `totalAllocations`: heap allocations during the window (0 = pool-only)
- `peakMemoryBytes`: peak resident memory during the window
- `collisionChecksTotal`: total collision detection operations
- `collisionHitsTotal`: collisions actually detected

Drivers that cannot measure a metric internally should return `-1` for that field. The conformance suite will fall back to external measurement where possible.

### `replay` (NEW in v1)

Returns a compact deterministic replay log for the last N ticks.

```json
{"cmd":"replay","ticks":1000}
```

Response:

```json
{
  "ok": true,
  "replay": {
    "seed": 42,
    "startTick": 0,
    "endTick": 1000,
    "inputSequence": "AQABAAAAAAA...",
    "stateChecksum": "sha256:a1b2c3d4..."
  }
}
```

Required fields:
- `seed`: the init seed
- `startTick`: first tick in the replay window
- `endTick`: last tick in the replay window
- `inputSequence`: base64-encoded packed input sequence (1 byte per tick: steer+1 in bits 0-1, boost in bit 2, jump in bit 3)
- `stateChecksum`: SHA-256 hex digest of all canonical GameState JSON in the window concatenated with `\n`

The replay contract: given the same seed and inputSequence, a fresh driver process must produce an identical stateChecksum. This is the basis for AC26 (Deterministic Replay Accuracy).

### `challenge` (NEW in v1)

Activates a specific stress-test scenario for the conformance suite. This is NOT a hint system â€” it's a way to configure initial conditions that would be impractical to reach through normal play.

```json
{"cmd":"challenge","name":"dense_field","params":{"obstacleCount":100}}
```

The `challenge` command must be called AFTER `init` and BEFORE any `step`. It modifies the initial state for stress testing only. The conformance suite uses it for:

- `dense_field`: spawns N obstacles in the visible area (AC18)
- `high_speed`: starts skier at max boost speed (AC17)
- `evasion_course`: spawns monster at distance 100m instead of 2000m (AC19)
- `crash_gauntlet`: spawns obstacles in a line forcing 50 consecutive crashes (AC21)

Response:

```json
{"ok":true,"state": <GameState>}
```

Drivers that receive an unknown challenge name respond with:

```json
{"ok":false,"error":"unknown challenge: foo"}
```

Challenges are optional for drivers. A driver that doesn't implement challenges can return an error and the conformance suite will attempt to reach the same conditions through normal play (which may fail, causing those ACs to be untestable â€” not failed, but skipped with a note).

## GameState (Enhanced)

```json
{
  "skier": { "x": 0, "y": 0, "speed": 0, "mode": "skiing" },
  "distanceM": 0,
  "style": 0,
  "obstacles": [
    { "type": "tree", "x": 12, "y": 96, "width": 2.0, "height": 3.0 }
  ],
  "monster": null,
  "gameOver": false,
  "tick": 0,
  "events": [],
  "quality": {
    "tickNanos": 2300000,
    "collisionChecks": 15,
    "collisionHits": 2,
    "activeObjects": 42,
    "memoryBytes": 45000000
  }
}
```

### New/changed fields

**obstacles[].width, obstacles[].height** (NEW, optional):
- Obstacle dimensions in the same coordinate space as skier
- Enables pixel-perfect collision testing (AC6 quality score, AC17)
- If absent, the suite assumes default hitboxes (tree: 2x3, rock: 3x2, etc.)

**events** (NEW, optional):
- Array of event strings for the current tick
- Possible values: `"collision"`, `"crash"`, `"recovery"`, `"airborne"`, `"landing"`, `"style_gain"`, `"style_loss"`, `"monster_spawn"`, `"monster_eat"`, `"monster_flee"`, `"near_miss"`, `"particle_spawn"`, `"screen_shake"`, `"color_grading_active"`
- Enables precise timing measurement (AC23 input responsiveness, AC24 near-miss detection)

**quality** (NEW, optional):
- Per-tick performance metrics
- `tickNanos`: wall-clock time for this tick's logic
- `collisionChecks`: collision detection operations this tick
- `collisionHits`: collisions detected this tick
- `activeObjects`: total active game objects (obstacles + monster + skier + particles)
- `memoryBytes`: current memory usage

If `quality` is absent, the conformance suite measures externally using process-level timing and memory inspection.

## Determinism Requirement

Unchanged from v0: for any seed and command/input sequence, two independent driver processes must emit identical GameState streams after canonical JSON serialization.

v1 adds: the `replay` command's `stateChecksum` must match across processes for the same seed and input sequence.

## Error Response

Unchanged from v0:

```json
{"ok":false,"error":"unsupported command"}
```

An error response, invalid JSON, EOF, timeout, or process crash causes the active conformance check to fail.
